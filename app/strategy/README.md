# Strategy Module

## 개요
전략 모듈은 자동매매 시스템에서 **어떤 종목을 대상으로**, **어떤 시점에 매수/매도할지** 판단하는 역할을 담당한다.

본 시스템의 전략은 단일 단계가 아니라 아래와 같이 **2단계 구조**로 운영한다.

1. **타깃 종목 선정**
2. **실시간 매매 실행**

즉, 전 종목을 실시간 감시하는 구조가 아니라,  
과거 데이터와 사전 조건을 기반으로 **감시할 종목군을 먼저 좁힌 뒤**,  
해당 종목에 대해서만 장중 실시간 데이터를 받아 전략을 실행한다.

---

## 전략 운영 방식

### 1. 타깃 종목 선정
장 시작 전 또는 장중 특정 주기마다 과거 시장 데이터를 기반으로  
전략 실행 대상 종목을 선정한다.

이 단계의 목적은 다음과 같다.

- 전 종목 실시간 구독 방지
- 전략 감시 대상 축소
- 불필요한 데이터 처리량 감소
- 실매매 대상 종목군 집중 관리

타깃 선정에 사용할 수 있는 조건 예시는 다음과 같다.

- 최근 거래대금 상위 종목
- 최근 변동성 확대 종목
- 이동평균 추세 정배열 종목
- 특정 기간 고점/저점 돌파 후보
- 거래량 급증 종목
- 갭 상승/하락 종목
- 사용자 정의 watchlist

선정 결과는 당일 전략 감시 대상 universe로 사용한다.

---

### 2. 실시간 시장 데이터 구독
타깃 종목이 선정되면, 해당 종목에 대해서만 브로커 WebSocket을 통해  
실시간 체결가/호가/거래량 데이터를 수신한다.

이 단계의 목적은 다음과 같다.

- 선정 종목만 집중 감시
- 진입/청산 조건의 실시간 평가
- 주문 트리거 발생 시점 포착

실시간 데이터는 전 종목 대상이 아니라  
**선정된 종목 단위 subscribe 방식**으로 운용한다.

---

### 3. 장중 전략 감시
전략 워커는 수신한 실시간 데이터를 바탕으로  
각 종목의 매매 조건을 계속 평가한다.

평가 대상은 단순 가격뿐 아니라 다음 요소를 포함할 수 있다.

- 현재가
- 시가 / 고가 / 저가
- 체결량 / 누적 거래량
- 호가 잔량
- 분봉 기준 가격 흐름
- 실시간 갱신 지표
- 현재 보유 포지션 상태
- 당일 이미 주문한 이력

이 단계에서 전략은 **매수 / 매도 / 관망(HOLD)** 중 하나의 판단을 생성한다.

---

### 4. 전략 시그널 생성
전략 조건이 충족되면 전략 시그널을 생성한다.

예시:
- `BUY`
- `SELL`
- `HOLD`

전략 모듈은 직접 브로커 주문을 호출하지 않고,  
우선 **시그널 생성** 또는 **주문 요청 생성 요청**까지만 담당한다.

이렇게 분리하는 이유는 다음과 같다.

- 전략과 주문 실행 책임 분리
- 중복 주문 방지 로직 분리
- 리스크 체크 계층 분리
- 주문 장애 발생 시 전략 계층 오염 방지

---

### 5. 주문 요청 생성
전략 시그널이 발생하면 실제 주문 전송 전에  
주문 생성 계층에서 추가 검증을 수행한다.

검증 항목 예시:
- 이미 동일 종목 주문이 진행 중인지
- 현재 보유 수량과 충돌하는지
- 주문 가능 시간인지
- 주문 가능 수량/자금이 충분한지
- 동일 전략의 중복 진입인지
- 1일 최대 진입 횟수를 초과했는지

검증 통과 시 주문 요청 레코드를 생성하고,  
이후 주문 워커가 실제 브로커 API 주문을 수행한다.

---

### 6. 주문 실행 및 상태 추적
주문 실행은 전략 모듈이 아니라 기존 주문 시스템이 담당한다.

역할 분리는 다음과 같다.

- **전략 모듈**
  - 종목 선정
  - 실시간 감시
  - 시그널 생성

- **주문 모듈**
  - 주문 요청 생성
  - 브로커 API 호출
  - 주문 상태 관리

- **주문 추적 모듈**
  - ACCEPTED / FILLED / PARTIAL_FILLED / CANCELED / FAILED 추적

즉, 전략은 “사야 하는가 / 팔아야 하는가”를 판단하고,  
주문 시스템은 “실제로 주문을 넣고 끝까지 추적”한다.

---

## 전략 설계 원칙

### 1. 전 종목 실시간 감시는 지양
국내 주식 전 종목을 실시간 스트리밍으로 감시하는 방식은  
데이터량과 처리 비용이 크므로 현재 시스템 범위에서는 채택하지 않는다.

본 시스템은 **선정 종목 기반 감시 방식**을 기본으로 한다.

---

### 2. 전략은 2단계로 분리
전략은 다음 두 층으로 구분한다.

- **선정 전략**
  - 오늘 어떤 종목을 볼 것인가

- **실행 전략**
  - 지금 이 종목을 살 것인가 / 팔 것인가

이 구조를 통해 전략을 더 명확하게 분리하고 관리한다.

---

### 3. 전략과 주문 실행은 분리
전략은 시그널을 생성하되,  
브로커 주문 API를 직접 호출하지 않는다.

주문 실행 책임은 별도 주문 계층으로 분리하여  
안정성과 추적 가능성을 확보한다.

---

### 4. 전략은 상태를 인지해야 함
실시간 전략은 단순 조건식만으로 동작하지 않는다.  
반드시 다음 상태를 함께 고려해야 한다.

- 현재 보유 여부
- 평균 매입 단가
- 미체결 주문 존재 여부
- 당일 진입 횟수
- 손절/익절 조건 충족 여부
- 장 운영 상태

즉, 전략은 가격만 보는 함수가 아니라  
**시장 상태 + 계좌 상태 + 주문 상태를 함께 해석하는 모듈**이어야 한다.

---

### 5. 원시 tick 전체 저장은 필수가 아님
실시간 전략 운영 초기 단계에서는  
모든 원시 체결 데이터를 영구 저장하지 않는다.

우선은 다음 위주로 운용한다.

- 현재 최신 가격 상태
- 전략 판단에 필요한 최소 지표
- 필요한 분봉 집계 데이터
- 주요 이벤트 로그

저장 비용과 복잡도를 줄이고,  
전략 검증에 필요한 최소 데이터만 유지한다.

---

## 전체 실행 흐름

1. 과거 데이터 수집
2. 조건 검색으로 당일 타깃 종목 선정
3. 선정 종목 실시간 데이터 구독
4. 장중 전략 조건 지속 평가
5. 매수/매도 시그널 생성
6. 주문 요청 생성
7. 주문 워커가 실제 주문 실행
8. 주문 추적 워커가 상태 추적
9. 체결/취소/실패 결과를 전략 상태에 반영

---

## 기대 효과
이 구조를 통해 다음을 기대한다.

- 전 종목 실시간 감시 부담 완화
- 전략 처리 범위 축소
- 주문 계층과 전략 계층의 명확한 분리
- 전략 교체 및 확장 용이
- 백테스트 구조와 실매매 구조의 자연스러운 연결

---

## 향후 확장 방향
향후 전략 모듈은 다음 방향으로 확장할 수 있다.

- 복수 전략 동시 실행
- 장 시작 전 후보군 자동 생성
- 장중 재선정 로직
- 분봉 실시간 집계
- 포지션 관리 전략 분리
- 손절/익절/트레일링 스탑 정책 추가
- 전략별 자본 배분 정책 추가
- 전략별 성과 기록 및 비교

---

## 결론
본 시스템의 전략은 다음 흐름으로 운영한다.

> **과거 데이터 기반 타깃 종목 선정 → 선정 종목 실시간 감시 → 시그널 생성 → 주문 시스템으로 전달**

이 구조를 기반으로  
백테스트에서 검증된 전략을 실매매 구조로 자연스럽게 이관할 수 있도록 설계한다.
